# colorlight-ecp5-tools

RTL utilities and test benches for the **Colorlight 5A-75B v8.0** (Lattice ECP5-25F, speed-6).

Toolchain: [Yosys](https://github.com/YosysHQ/yosys) + [nextpnr-ecp5](https://github.com/YosysHQ/nextpnr) + [Project Trellis](https://github.com/YosysHQ/prjtrellis) + [openFPGALoader](https://github.com/trabucayre/openFPGALoader).

---

## Board pinout (v8.0)

| Signal     | Pin | Notes                          |
|------------|-----|--------------------------------|
| CLK 25 MHz | P6  | Onboard oscillator             |
| LED        | T6  | Active-low, green              |
| Button     | R7  | Active-low                     |
| NTSC sync  | P5  | J4 HUB75 connector             |
| NTSC video | R3  | J4 HUB75 connector             |
| OLED SCL   | E4  | J1 B0, via 74HC245T buffer     |
| OLED SDA   | D3  | J1 R1, via 74HC245T buffer     |

Programming: SVF via FT232H JTAG.

```bash
openFPGALoader -c ft232 build/foo.svf          # SRAM
openFPGALoader -c ft232 --write-flash foo.bit  # NOR flash
```

## What's here

### `rtl/ntsc_grey.v` — greyscale NTSC display

Dual-clock 320×240 greyscale NTSC output with no external DAC beyond a 2-resistor
voltage divider (sync: 560 Ω, video: 220 Ω).

**Architecture:**
- `clk_25` (25 MHz) — NTSC timing counters, BRAM reads, pixel tracking
- `clk_800` (800 MHz from PLL) — free-running 5-bit PWM counter, video pin output

Each 25 MHz pixel slot = 32 × 800 MHz ticks, giving 32 grey levels. The CRT
integrates luminance over the pixel period, so PWM at 800 MHz is invisible as
flicker and reads as analogue brightness.

**4-bit storage, 5-bit PWM:**
Four 9600-byte bitplane ROMs (40 bytes × 240 rows, 8 pixels/byte) store 4-bit pixel
values (0–15). Intensity maps to `{pixel_int, 1'b0}` → even PWM levels 0, 2, 4 … 30.

**Half-step blending via CDC latency:**
A 16-stage shift register in the 800 MHz domain delays the clock-domain crossing by
exactly half a pixel period (16 ticks = 20 ns). At every pixel boundary the PWM
counter runs 16 ticks with the previous pixel's intensity then 16 ticks with the new,
blending adjacent values and producing odd intermediate levels. This gives smooth
perceptual transitions between the 16 stored levels with zero extra logic — the delay
pipeline is all FFs.

Pixel data is stable for 40 ns (one full 25 MHz period) so the 20 ns delay still
leaves a safe 20 ns CDC capture window.

**BRAM never sees 800 MHz.** All memory access is in the 25 MHz domain. The 800 MHz
domain contains only a 5-bit counter, a comparator, and the shift register — well
within ECP5 routing capability at that frequency.

```
clk_25  ──► NTSC counters ──► BRAM read ──► pixel_int ──► intensity ──┐
                                                                        │ 16-FF pipeline
clk_800 ──► pwm_cnt[4:0] ──► (pwm_cnt < intensity_800) ──► video_pin ◄┘
```

### `rtl/pll_200.v` — ECP5 PLL

`EHXPLLL` configured for:
- VCO = 25 × 32 = 800 MHz
- `CLKOP` (÷1) → 800 MHz (PWM / video output)
- `CLKOS` (÷32) → 25 MHz (NTSC + BRAM, phase-locked to crystal)

### `rtl/ntsc_framebuf.v` — 1-bit NTSC framebuffer

Single-clock 320×240 @ 1 bpp, three pre-rendered bitmaps (RUN/PASS/FAIL) for
test-result display on a composite CRT. Used as the display front-end for the
Spirix floating-point self-test harness.

### `rtl/ntsc_text.v` — text overlay

Composites an 8×8 font ROM over the framebuffer output. Font generated by
`tools/gen_font_rom/` (Oxanium ExtraLight via fontdue).

### `rtl/ssd1306_oled.v` / `ssd1306_i2c.v` — SH1106/SSD1306 OLED

Bit-bang I2C controller and display driver for 128×64 monochrome OLEDs.
Shows four 32-bit registers as horizontal bit-bar graphs.

**Gotcha:** drive SDA **low** during the ACK clock (not release/high). The
74HC245T push-pull buffer fights the SH1106 open-drain ACK and the display
stops responding after the first frame if you get this wrong.

### `rtl/top_pin_scan.v` — pin continuity scanner

Sequentially drives each I/O pin low and probes all others. Used to map out
direct-wired hardware where pin assignments are unknown. Results displayed on
the OLED.

### `src/main.rs` — `img2mem` image converter

Converts any image to four greyscale bitplane `.mem` files for `$readmemh`.

```
cargo run --release -- input.jpg build/
# writes build/grey{0,1,2,3}.mem
```

Resizes to 320×240 (Lanczos3), converts to luma, quantises to 4 bits, packs
8 pixels per byte (MSB = leftmost pixel), and writes one file per bitplane.

## Building the greyscale demo

You need a JPEG or PNG at any size named `test.jpg` in the project root.

```bash
SEED=4 bash scripts/build_grey.sh          # build only
SEED=4 bash scripts/build_grey.sh --program  # build + program SRAM
SEED=4 bash scripts/build_grey.sh --flash    # build + write NOR flash
```

Placement seed 4 gives the best routing on the ECP5-25F for this design.
nextpnr will report timing failures against the 800 MHz constraint — this is
expected. nextpnr's static timing is approximately **2–2.5× pessimistic** on
ECP5-25F speed-6 silicon. Measured silicon Fmax for the 800 MHz PWM logic
is well above 800 MHz.

## Notes on ECP5 timing margins

`nextpnr-ecp5` static timing analysis is consistently 2–2.5× below actual
silicon Fmax on ECP5-25F speed-6 (measured across many designs including
adders, multipliers, BLAKE3, and LFSR chains). A design that estimates
400 MHz typically clears 800 MHz on the real device. This is useful to know
when you are using a global clock that exceeds the tool's confidence.
